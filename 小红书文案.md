# 给 LLM 看 C 代码？这个工具帮你省 60% Token

---

不是我夸张，做嵌入式和 C/C++ 的同学应该都懂这种痛——

你兴冲冲把项目代码丢给 Claude 或者 ChatGPT 让它帮你分析，结果它先读了几百行 `#ifdef PRODUCT_A` 的兼容代码，再啃几百行 `#ifdef PRODUCT_B` 的冗余分支，Token 哗哗地烧，真正有用的分析还没开始。

**MacroPruner-Ctx 就是来解决这个的。**

---

## 它做了什么？

一句话：**在 LLM 读到 C/C++ 文件之前，自动把不属于当前目标的 `#ifdef` 代码块删掉。**

比如你正在开发 PRODUCT_A，代码里嵌套了这样的东西：

```c
#ifdef PRODUCT_A
  init_feature_a();    // ← 保留
#else
  init_feature_b();    // ← 删掉
#endif
```

工具会识别出 PRODUCT_A 是活跃宏，把 `#else` 分支直接移除。实测下来，**一个典型的多产品嵌入式项目，单文件压缩率能达到 40%-60%。**

也就是说，同样的 Token 预算，你喂给 LLM 的有效代码量翻了一倍多。

---

## 三段式压缩漏斗

MacroPruner-Ctx 采用递进式压缩策略，从粗到细层层瘦身：

**Stage 1：宏裁剪（Macro Pruning）**
- 支持 `#ifdef` / `#ifndef` / `#else` / `#elif` 完整嵌套，不限深度
- 两个模式：**物理删除**（最省 Token）和 **虚拟折叠**（保留行号标记，方便调试）
- 从 `compile_commands.json` 自动提取 `-D` 宏，不用手动写配置
- 单文件压缩 40-60%

**Stage 2：骨架化（Skeletonization）**  
- 剥离所有函数体，只保留 struct/enum/typedef 定义、宏常量和函数签名
- 适合快速了解模块接口、生成 API 文档
- 相比完整代码再省 70-90%

**Stage 3：依赖图（Dependency Graph）**
- 解析 `#include` 树，一次调用获取跨文件上下文
- 目标文件：完整裁剪 → 依赖文件：裁剪 + 骨架化
- LLM 看到的不是孤立单文件，而是完整的模块视
- 可配置最大 include 深度，Token 可控

---

## 四个 MCP 工具，接入任意 Agent

1. **`read_c`** — 单文件宏裁剪，最常用的入口
2. **`read_c_skeleton`** — 骨架化浏览，只看接口不看实现
3. **`read_c_with_deps`** — 多文件上下文，目标全量 + 依赖骨架
4. **`apply_patch`** — LLM 生成 unified diff，精准写回原文件，最小改动风险

打包成 MCP Server，一行命令接入 Claude Desktop、Hermes 等 Agent。**27 个测试用例全部通过**，覆盖 pruner、skeletonizer、dep_graph、E2E。

## 怎么用？

```bash
git clone https://github.com/shouchengcheng/macropruner-ctx.git
cd macropruner-ctx
python3 -m venv .venv && source .venv/bin/activate
pip install mcp
```

Agent 里调用：

```
read_c(file_path="src/main.c", target="PRODUCT_A",
       compile_db="/path/to/compile_commands.json")
```

LLM 看到修剪后的代码并给出修改建议后，调用：

```
apply_patch(file_path="src/main.c", diff="...")
```

通过 unified diff 将改动精准写回原文件，风险最小化。

---

## 适合谁？

- 嵌入式开发 / 多产品线维护，代码里 `#ifdef` 满天飞的
- 用 LLM 做代码审查、重构、文档生成的
- 想把 LLM 深度集成到 C/C++ 工作流的
