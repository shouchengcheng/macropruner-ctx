1. 一句话根因

pruner_core.py 的 4 个 _handle_* 兜底分支用 return line，但 line 在这些方法作用域内从未定义（签名只有 line_num，调用方 process_line 也没把 line 传进去）。任何含有"孤立 #elif / #else / #endif"或"不平衡 #endif"的源码都会触发 NameError: name 'line' is not defined。

2. 调用链与代码定位

完整调用链（MCP 工具 → Python 类）：

mcp_server.py::read_c / read_c_skeleton / read_c_with_deps
  └─ _prune_file(file_path, target, compile_db, mode)
       └─ pruner.prune(source)                        ← pruner_core.py:187
            └─ for line_num, line in enumerate(...)  ← pruner_core.py:200
                 └─ process_line(line, line_num)      ← pruner_core.py:80
                      └─ directive_match.group(...)
                      └─ self._handle_if(directive, condition, line_num)        ← 注意：没传 line
                      └─ self._handle_elif(condition, line_num)                  ← 同上
                      └─ self._handle_else(line_num)                              ← 同上
                      └─ self._handle_endif(line_num)                             ← 同上

process_line() 在 pruner_core.py:88-95 调三个 _handle_* 方法时，只传了 line_num，没有传 line。三个 _handle_* 方法签名也没有 line 参数。但每个方法在"栈空 / 指令类型不符"的兜底分支都写了 return line —— line 在那个作用域里根本不存在。

4. 为什么现有测试没抓到

所有 7 个测试都只覆盖"语法正确嵌套"的正常路径，不覆盖以下边界：
  • 孤立的 #elif（无前置 #if）
  • 孤立的 #else（无前置 #if）
  • 孤立的 #endif（无前置 #if）
  • 多个 #endif 紧跟（第一个弹空栈后第二个进入兜底）
  • 嵌套深度失衡（#if A → #else → #if B → 缺少 #endif B 直接 #endif A）

固件 SDK（尤其 ws63、RTL8720）大量使用 #if 0 块、跨文件 #ifdef 嵌套、和 release/debug 切换宏，几乎必然会在某些真实文件上触发孤立 #endif。wm_fota.c 就是其中之一（31885 字节，含 #if 0 块、嵌套 #ifdef，本会话就被它卡住）。

5. 修复方案

三种思路，按推荐度排序：

方案 A（推荐）：给 3 个方法加 line 参数，调用方一并传入

优点：保留原代码意图（"兜底时把原行返回"）；最小行为变化；不引入静默错误。

# pruner_core.py
# 修改 1: process_line 调用处加 line 参数（line 88-95）
def process_line(self, line: str, line_num: int) -> str:
    ...
    if directive in ("ifdef", "ifndef"):
        return self._handle_if(directive, condition, line, line_num)   # +line
    elif directive == "elif":
        return self._handle_elif(condition, line, line_num)            # +line
    elif directive == "else":
        return self._handle_else(line, line_num)                       # +line
    elif directive == "endif":
        return self._handle_endif(line, line_num)                      # +line

# 修改 2: 4 个方法签名加 line 参数
def _handle_if(self, directive: str, condition: str, line: str, line_num: int) -> str:
    # 该方法无 return line，无需改 body
    ...

def _handle_elif(self, condition: str, line: str, line_num: int) -> str:
    if not self.stack:
        return line                                                    # 现在合法
    top_block = self.stack[-1]
    if top_block.directive not in ("ifdef", "ifndef", "elif"):
        return line                                                    # 现在合法
    ...

def _handle_else(self, line: str, line_num: int) -> str:
    if not self.stack:
        return line                                                    # 现在合法
    top_block = self.stack[-1]
    if top_block.directive not in ("ifdef", "ifndef", "elif"):
        return line                                                    # 现在合法
    ...

def _handle_endif(self, line: str, line_num: int) -> str:
    if not self.stack:
        return line                                                    # 现在合法
    ...

方案 B（保守）：3 个 return line 全改成 return None

优点：改动最小，1 行 5 处替换。
缺点：和"原样返回 line"的原意不同；静默丢行可能掩盖真实语法问题。

# pruner_core.py - pr 级别替换即可
sed -i 's|        return line$|        return None|g' pruner_core.py
# 然后人工确认 line 126/131/154/159/179 五处

方案 C（激进的正确方案）：兜底分支直接 raise ValueError

优点：让真实语法错误立刻暴露，避免静默坏数据。
缺点：会破坏"宽松解析"的设计意图；如果用户刻意写了孤立 #endif 当注释（罕见但有），会失败。

def test_orphan_elif():
    """孤立 #elif (stack 为空) - 不应抛 NameError"""
    p = PrunerCore(active_macros={}, mode=PrunerMode.PHYSICAL_DELETION)
    r = p.process_line("#elif FOO", 1)
    # 方案 A: 返回原 line; 方案 B: 返回 None; 方案 C: 抛 ValueError
    assert r is None or r == "#elif FOO"

def test_orphan_else():
    """孤立 #else - 不应抛 NameError"""
    p = PrunerCore(active_macros={}, mode=PrunerMode.PHYSICAL_DELETION)
    r = p.process_line("#else", 1)
    assert r is None or r == "#else"

def test_orphan_endif():
    """孤立 #endif - 不应抛 NameError"""
    p = PrunerCore(active_macros={}, mode=PrunerMode.PHYSICAL_DELETION)
    r = p.process_line("#endif", 1)
    assert r is None or r == "#endif"

def test_double_endif():
    """连续两个 #endif - 第二个应进入兜底"""
    p = PrunerCore(active_macros={}, mode=PrunerMode.PHYSICAL_DELETION)
    p.process_line("#ifdef FOO", 1)
    p.process_line("#endif", 2)
    r = p.process_line("#endif", 3)  # stack 已空
    assert r is None or r == "#endif"

def test_unbalanced_real_world():
    """真实场景: #if 0 块 + 不平衡 #endif + 孤立 #else"""
    src = """line1
#if 0
removed
#else
kept
"""
    p = PrunerCore(active_macros={}, mode=PrunerMode.PHYSICAL_DELETION)
    out = p.prune(src)   # 修前会抛 NameError，修后应返回有效结果
    assert "line1" in out
    assert "kept" in out
