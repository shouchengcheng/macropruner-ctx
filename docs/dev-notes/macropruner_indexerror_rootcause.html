MacroPruner-Ctx IndexError 根因排查报告

日期：2026-06-12 状态：已修（3 处 patch + 全套测试通过） 关联报告：macropruner_nameerror_rootcause.html（历史 NameError，已不存在）
TL;DR：源码里 #if ... \<NL> defined(...) 的反斜杠续行被 pruner_core 忽略，截断的 condition 传给 expr_eval._parse_unary_atom 后越界读 self.tokens[self.pos]，抛 IndexError。修复：3 处 (expr_eval.py 越界 raise ValueError + pruner_core.py 兜底扩到 IndexError + mcp_server.py FileNotFoundError hint 区分场景)。

1. 一句话根因

pruner_core 的 prune() 逐行扫描预处理指令，不识别 \<NL> 续行。当遇到 #if EXPR_A || EXPR_B || \<NL> EXPR_C 这种条件时，EXPR_C 被留在下一行不进入当前 condition 字符串；结果传给 expr_eval 的 condition 是被截断的 ... ||（末尾的不完整运算符），parser 走到 _parse_unary_atom 期望读下一个 token 时 self.pos 已经越界，触发 IndexError: list index out of range。

2. 调用链与代码定位

mcp_server.py:read_c
  → _prune_file
    → inst.prune (regex backend, backends/regex_backend.py)
      → PrunerCore.process_line  (pruner_core.py:96)
        → _handle_if
          → self.evaluate_condition (pruner_core.py:139)
            → ExpressionEvaluator.evaluate (expr_eval.py:131)
              → parse_expression → ... → _parse_unary_atom (expr_eval.py:294)
                → kind, val = self.tokens[self.pos]   # ← IndexError here

2.1 触发源码模式（实测）

文件：/middleware/ble/.c:463

#if defined(AUXSDK_SUPPORT_MULTI) || defined(AUXSDK_SUPPORT_HEATPUMPS) ||      \
    defined(AUXSDK_OVERSEAS) || defined(AUXSDK_SUPPORT_MVN)

传给 expr_eval 的 condition 是被截断的：

defined(AUXSDK_SUPPORT_MULTI) || defined(AUXSDK_SUPPORT_HEATPUMPS) ||

expr_eval tokenize 后：

DEF AUXSDK_SUPPORT_MULTI ) || DEF AUXSDK_SUPPORT_HEATPUMPS ) ||

parser 解析完第二个 defined(...)，期待下一个 token（通过 _parse_unary_atom 读），但 self.pos 已经到了 token 列表末尾。越界 self.tokens[self.pos] 抛 IndexError。

2.2 IndexError 漏到 MCP 层的链路

pruner_core.py:80 旧版 evaluate_condition 只 catch ValueError：

try:
    result = self._evaluator.evaluate(condition)
    return bool(result)
except ValueError:        # ← 只 catch ValueError，IndexError 漏出
    return False

IndexError 一路冒到 mcp_server.py 的 read_c 兜底 except Exception as e: return format_error(e)，被包装成 [ERROR] IndexError: list index out of range 返回给 LLM。

4. 为什么现有测试没抓到

现有 test_pruner_realistic.py 10 个 case 全是基于简单 #if defined(X) 模式（单 token、单 identifier、IS_ENABLED 宏等），没有"行末 \<NL> 续行 + 多重 ||"的 case。仓库内大部分 C 源也不这么写（cond 都在单行内），但 / / 习惯把长条件用 \<NL> 续到下一行，所以这个故障在普通工程上很难浮现，在 项目上稳定出现。

这是测试覆盖盲区，**建议补一个回归测试**（见 §7）。

5. 修复方案（3 处）

#文件行改动行为
1expr_eval.py294_parse_unary_atom 越界 raise ValueError("unexpected end of expression")错误模型与其他 parser 函数（_parse_defined、_parse_identifier_or_call）一致；上层能 catch
2pruner_core.py80evaluate_condition catch 扩到 (ValueError, IndexError)一个 #if 解析失败不再炸整个 prune 流程（"best-effort, never crash"）
3mcp_server.pyread_c FileNotFoundError 兜底按 str(e) 内容分 3 种 hint区分"文件不存在"/"compile_db 缺失"/其他，不再统一误导成"调 bootstrap_config"

3 处改动都留了 .bak.20260612_143436 备份。

5.1 修复 1：expr_eval.py:294

# 修复前
def _parse_unary_atom(self):
    kind, val = self.tokens[self.pos]  # ← 越界抛 IndexError

# 修复后
def _parse_unary_atom(self):
    if self.pos >= len(self.tokens):
        raise ValueError("unexpected end of expression")
    kind, val = self.tokens[self.pos]

理由：该模块其他 parser 函数（_parse_defined）都用 if self.pos >= len(self.tokens) 守一下，_parse_unary_atom 漏了。补上保证错误模型一致（都是 ValueError）。

5.2 修复 2：pruner_core.py:80

# 修复前
try:
    result = self._evaluator.evaluate(condition)
    return bool(result)
except ValueError:
    return False

# 修复后
try:
    result = self._evaluator.evaluate(condition)
    return bool(result)
except (ValueError, IndexError):
    return False

理由：单点失败不应该毁掉整个 prune。注释里写明这是"best-effort, never crash"契约——与原 ValueError catch 的设计意图一致，只是把 IndexError 也纳入。理论上也考虑过扩到 Exception，但那样会吞掉 AttributeError 之类的真 bug 反而难以发现。

5.3 修复 3：mcp_server.py:read_c FileNotFoundError 兜底

# 修复前
except FileNotFoundError as e:
    return FatalError(
        str(e),
        hint="verify the path exists, or call bootstrap_config() to auto-generate "
             "the config, or drop a .macroprunerrc with 'compile_db = ...'",
    ).formatted()

# 修复后
except FileNotFoundError as e:
    msg = str(e)
    if "compile_db" in msg or "compile_commands.json" in msg:
        hint = "compile_db is missing or unreadable — pass a valid absolute path ..."
    elif "Cannot resolve file path" in msg:
        hint = "the source file does not exist at that path — check spelling, case, ..."
    else:
        hint = "verify the path exists, or call bootstrap_config() ..."
    return FatalError(msg, hint=hint).formatted()

理由：_prune_file 抛的 FileNotFoundError 有 3 种来源（用户路径错 / compile_db 找不到 / 后端内部 IO），hint 写死成"调 bootstrap_config"对前两种都是误导。

7. 建议新增的测试用例（回归）

在 test_pruner.py 末尾加：

def test_truncated_condition_from_line_continuation():
    """A #if condition truncated by `\\` should not crash pruner.

    Reproduces the IndexError that surfaced on .c:463 where
    `defined(A) || defined(B) || \\\\\\    defined(C)` was passed to
    expr_eval without the line-continuation being preprocessed, leaving
    `... ||` with no right operand. The parser walked past the end of
    the token stream and raised IndexError.
    """
    source = textwrap.dedent('''
        #if defined(A) || defined(B) ||      \\\\
            defined(C)
        void active() {}
        #else
        void inactive() {}
        #endif
    ''')
    p = PrunerCore({'A': None, 'B': None, 'C': None}, PrunerMode.PHYSICAL_DELETION)
    # Must not raise IndexError; inactive branch should be pruned.
    pruned = p.prune(source)
    assert 'void active()' in pruned
    assert 'void inactive()' not in pruned

9. 与 NameError 报告的关系

macropruner_nameerror_rootcause.html 描述的是 pruner_core.py:126/131/154/159/179 共 5 处 return line 引用未定义变量的旧 bug。那个 bug 在 2026-06-12 同步 review 时发现已不存在——当前代码那 6 处 return line 都引用合法的函数参数。NameError 报告保留为历史档案用，不要按它的根因去查当前 bug。

本次报告的 IndexError 才是当前实际故障模式，已修复并验证。
